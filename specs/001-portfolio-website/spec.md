# 기능 명세서: 개발자 포트폴리오 웹사이트

**기능 브랜치**: `001-portfolio-website`  
**생성일**: 2025-11-11  
**상태**: 초안  
**입력**: 사용자 설명: "이 프로젝트는 내 이력을 나타내는 포트폴리오야. 이 앱을 주로 보는 사람들은 기업이거나 면접관들이 보게될거야. 그렇기 때문에 이 앱을 통해서 내가 돋보일 수 있어야함. 앱은 하나의 경로(/)만 가지는 싱글페이지 웹 사이트가 될거야."

## 사용자 시나리오 및 테스트 _(필수)_

### 사용자 스토리 1 - 포트폴리오 핵심 정보 확인 (우선순위: P1)

채용 담당자 또는 면접관이 웹사이트를 방문하여 개발자의 기본 정보, 기술 스택, 연락처를 즉시 파악할 수 있습니다.

**이 우선순위의 이유**: 이것은 포트폴리오 웹사이트의 가장 핵심적인 기능입니다. 방문자가 개발자에 대한 기본 정보를 빠르게 파악할 수 없다면 웹사이트의 목적을 달성할 수 없습니다. 이 스토리만으로도 최소한의 가치를 제공하는 MVP가 됩니다.

**독립 테스트**: 웹사이트에 접속하여 헤더, 히어로 섹션, About Me, Skills, Contacts 섹션이 모두 표시되고, 각 섹션의 정보가 읽을 수 있는지 확인함으로써 완전히 테스트할 수 있습니다. GitHub 링크 클릭 시 새 탭에서 GitHub 프로필이 열리는지 확인합니다.

**인수 시나리오**:

1. **Given** 방문자가 포트폴리오 웹사이트 URL을 브라우저에 입력했을 때, **When** 페이지가 로드되면, **Then** 헤더에 캐릭터 로고와 "Codeliner" 텍스트, 그리고 네비게이션 메뉴(About, Skills, Projects, Contact, Github Link)가 표시되어야 함
2. **Given** 페이지가 로드되었을 때, **When** 히어로 섹션을 확인하면, **Then** "Codeliner", "Front-End Developer" 타이틀, 소개 문구("코드를 통해 비즈니스 문제 해결과 효율적 설계를 고민하며 꾸준히 성장하고 있습니다."), 그리고 프로필 사진이 표시되어야 함
3. **Given** 페이지를 스크롤했을 때, **When** About Me 섹션을 확인하면, **Then** 2-3줄의 자기 소개 텍스트가 표시되어야 함
4. **Given** 페이지를 스크롤했을 때, **When** Skills 섹션을 확인하면, **Then** 기술 스택이 이미지와 텍스트가 포함된 그리드 형식으로 표시되어야 함
5. **Given** 페이지를 스크롤했을 때, **When** Contacts 섹션을 확인하면, **Then** 연락 문구와 GitHub 프로필로 이동하는 버튼 링크가 표시되어야 함
6. **Given** 헤더의 네비게이션 메뉴에서, **When** About, Skills, Projects, Contact 링크 중 하나를 클릭하면, **Then** 페이지가 해당 섹션으로 부드럽게 스크롤되어야 함
7. **Given** 헤더 또는 Contacts 섹션에서, **When** GitHub 링크를 클릭하면, **Then** 새 탭에서 개발자의 GitHub 프로필이 열려야 함

### 사용자 스토리 2 - 프로젝트 포트폴리오 탐색 (우선순위: P2)

채용 담당자 또는 면접관이 개발자가 진행한 사이드 프로젝트들을 캐러셀 UI를 통해 탐색하고 각 프로젝트의 세부 정보를 확인할 수 있습니다.

**이 우선순위의 이유**: 프로젝트 경험은 개발자의 실제 역량을 보여주는 중요한 요소이지만, 기본 정보 확인 다음 순서로 배치됩니다. 이 기능이 없어도 연락처를 통해 개발자와 소통할 수 있지만, 프로젝트를 보여줌으로써 지원자의 가치가 크게 향상됩니다.

**독립 테스트**: Projects 섹션에서 캐러셀의 좌우 네비게이션 버튼을 클릭하여 다른 프로젝트들을 탐색할 수 있는지, 각 프로젝트 카드에 프로젝트 이름, 설명, 사용 기술, 링크가 표시되는지 확인함으로써 테스트할 수 있습니다.

**인수 시나리오**:

1. **Given** 페이지를 스크롤하여 Projects 섹션을 확인했을 때, **When** 캐러셀 UI가 로드되면, **Then** 첫 번째 프로젝트가 표시되고 좌우 네비게이션 버튼(또는 인디케이터)이 보여야 함
2. **Given** Projects 섹션의 캐러셀이 표시되었을 때, **When** 다음 버튼을 클릭하면, **Then** 다음 프로젝트로 슬라이드되어 화면에 표시되어야 함
3. **Given** Projects 섹션의 캐러셀이 표시되었을 때, **When** 이전 버튼을 클릭하면, **Then** 이전 프로젝트로 슬라이드되어 화면에 표시되어야 함
4. **Given** 캐러셀에서 프로젝트 카드를 확인했을 때, **When** 각 프로젝트를 탐색하면, **Then** 프로젝트 이름, 간단한 설명, 사용된 기술 스택, 프로젝트 링크(GitHub, 라이브 데모 등)가 표시되어야 함
5. **Given** 프로젝트 카드에 외부 링크가 있을 때, **When** 링크를 클릭하면, **Then** 새 탭에서 해당 프로젝트의 저장소 또는 라이브 사이트가 열려야 함
6. **Given** 캐러셀에 여러 프로젝트가 있을 때, **When** 마지막 프로젝트에서 다음 버튼을 클릭하면, **Then** 첫 번째 프로젝트로 순환되어야 함(무한 루프)
7. **Given** 캐러셀에 여러 프로젝트가 있을 때, **When** 첫 번째 프로젝트에서 이전 버튼을 클릭하면, **Then** 마지막 프로젝트로 순환되어야 함

### 사용자 스토리 3 - 반응형 디자인 및 모바일 경험 (우선순위: P3)

방문자가 다양한 기기(데스크톱, 태블릿, 모바일)에서 포트폴리오 웹사이트를 원활하게 탐색할 수 있습니다.

**이 우선순위의 이유**: 많은 채용 담당자가 모바일 기기에서도 포트폴리오를 확인할 수 있지만, 초기 MVP에서는 데스크톱 경험에 집중하는 것이 더 중요합니다. 그러나 현대 웹 표준을 고려할 때 반응형 디자인은 필수적입니다.

**독립 테스트**: 브라우저의 개발자 도구를 사용하여 다양한 화면 크기(모바일, 태블릿, 데스크톱)에서 웹사이트를 확인하고, 모든 섹션이 적절하게 조정되고 읽을 수 있는지 테스트할 수 있습니다.

**인수 시나리오**:

1. **Given** 방문자가 모바일 기기(또는 좁은 화면)로 웹사이트에 접속했을 때, **When** 페이지가 로드되면, **Then** 헤더의 네비게이션 메뉴가 햄버거 메뉴로 전환되어야 함
2. **Given** 모바일 화면에서 헤더를 확인했을 때, **When** 햄버거 메뉴 아이콘을 탭하면, **Then** 네비게이션 메뉴가 드롭다운 또는 사이드바로 표시되어야 함
3. **Given** 모바일 화면에서 히어로 섹션을 확인했을 때, **When** 레이아웃을 확인하면, **Then** 텍스트와 프로필 사진이 수직으로 배치되어야 함
4. **Given** 모바일 화면에서 Skills 섹션을 확인했을 때, **When** 그리드를 확인하면, **Then** 기술 스택이 1-2열의 그리드로 조정되어 모바일에서 읽기 쉬워야 함
5. **Given** 태블릿 화면 크기로 웹사이트를 확인했을 때, **When** 모든 섹션을 탐색하면, **Then** 레이아웃이 화면 크기에 맞게 최적화되고, 콘텐츠가 잘리거나 겹치지 않아야 함
6. **Given** 데스크톱 화면에서 웹사이트를 확인했을 때, **When** 모든 섹션을 탐색하면, **Then** 전체 네비게이션 메뉴가 헤더에 표시되고, 콘텐츠가 넓은 화면을 효과적으로 활용해야 함

### 엣지 케이스

- 프로젝트가 하나도 없을 때 Projects 섹션은 어떻게 표시되나요? (최소 1개의 프로젝트는 있다고 가정하지만, 빈 상태 UI 고려)
- 기술 스택이 매우 많을 경우 (예: 20개 이상) Skills 섹션은 어떻게 표시되나요? (그리드가 자동으로 확장되거나 페이지네이션/더보기 버튼 추가)
- **프로필 사진이 로드되지 않을 때**: 히어로 섹션에 대체 이미지(placeholder)를 표시하여 시각적 일관성 유지
- **프로젝트 이미지가 로드되지 않을 때**: 프로젝트 카드에 대체 이미지(placeholder)를 표시
- **기술 아이콘이 로드되지 않을 때**: 기술 이름만 텍스트로 표시하거나 기본 아이콘 표시
- GitHub 링크가 유효하지 않을 때 시스템은 어떻게 처리되나요? (링크 유효성 검증 또는 에러 페이지 표시)
- 매우 긴 프로젝트 설명이 카드 레이아웃을 깨뜨릴 때 어떻게 처리되나요? (텍스트 말줄임 또는 "더보기" 기능)
- 네트워크 속도가 느릴 때 이미지 로딩은 어떻게 처리되나요? (로딩 스켈레톤 또는 스피너 표시)

## 요구사항 _(필수)_

### 기능 요구사항

**헤더 섹션**

- **FR-001**: 헤더는 반드시 좌측에 캐릭터 로고 이미지와 "Codeliner" 텍스트를 표시해야 함
- **FR-002**: 헤더는 반드시 우측에 네비게이션 링크(About, Skills, Projects, Contact, Github Link)를 표시해야 함
- **FR-003**: 네비게이션 링크는 반드시 클릭 시 해당 섹션으로 부드럽게 스크롤되어야 함
- **FR-004**: GitHub 링크는 반드시 클릭 시 새 탭에서 GitHub 프로필을 열어야 함
- **FR-005**: 헤더는 반드시 페이지 상단에 고정(sticky 또는 fixed)되어 스크롤 시에도 표시되어야 함

**히어로 섹션**

- **FR-006**: 히어로 섹션은 반드시 "Codeliner" (GitHub 이름)을 주요 제목으로 표시해야 함
- **FR-007**: 히어로 섹션은 반드시 "Front-End Developer" 부제목을 표시해야 함
- **FR-008**: 히어로 섹션은 반드시 소개 문구("코드를 통해 비즈니스 문제 해결과 효율적 설계를 고민하며 꾸준히 성장하고 있습니다.")를 표시해야 함
- **FR-009**: 히어로 섹션은 반드시 우측에 개발자의 프로필 사진을 표시해야 함
- **FR-010**: 히어로 섹션의 레이아웃은 반드시 텍스트 좌측, 이미지 우측으로 구성되어야 함

**About Me 섹션**

- **FR-011**: About Me 섹션은 반드시 개발자를 소개하는 2-3줄의 텍스트를 표시해야 함
- **FR-012**: About Me 텍스트는 반드시 읽기 쉬운 폰트 크기와 줄 간격으로 표시되어야 함

**Skills 섹션**

- **FR-013**: Skills 섹션은 반드시 기술 스택을 그리드 형식으로 표시해야 함
- **FR-014**: 각 기술 스택 항목은 반드시 기술 아이콘 이미지와 기술 이름 텍스트를 포함해야 함
- **FR-014-1**: 기술 아이콘은 반드시 외부 CDN(devicons, simpleicons 등)에서 로드되어야 함
- **FR-015**: 그리드는 반드시 화면 크기에 따라 열 수가 조정되어야 함 (반응형)

**Projects 섹션**

- **FR-016**: Projects 섹션은 반드시 프로젝트들을 캐러셀 UI로 표시해야 함
- **FR-017**: 캐러셀은 반드시 좌우 네비게이션 버튼(또는 화살표)을 제공해야 함
- **FR-017-1**: 캐러셀은 반드시 수동 조작으로만 작동해야 함 (자동 재생 미지원)
- **FR-018**: 각 프로젝트 카드는 반드시 프로젝트 이름, 설명, 사용 기술, 외부 링크를 포함해야 함
- **FR-019**: 캐러셀은 반드시 무한 루프 방식으로 작동해야 함 (마지막 → 첫 번째, 첫 번째 → 마지막)
- **FR-020**: 프로젝트 외부 링크는 반드시 클릭 시 새 탭에서 열려야 함
- **FR-021**: 캐러셀은 반드시 현재 표시 중인 프로젝트를 나타내는 인디케이터(점 또는 번호)를 표시해야 함

**Contacts 섹션**

- **FR-022**: Contacts 섹션은 반드시 연락을 유도하는 문구를 표시해야 함
- **FR-023**: Contacts 섹션은 반드시 GitHub 프로필로 이동하는 버튼 형식의 링크를 제공해야 함
- **FR-024**: GitHub 버튼은 반드시 클릭 시 새 탭에서 GitHub 프로필을 열어야 함

**반응형 디자인**

- **FR-025**: 웹사이트는 반드시 모바일 화면(< 768px)에서 햄버거 메뉴를 표시해야 함
- **FR-026**: 모바일에서 햄버거 메뉴는 반드시 클릭 시 네비게이션 메뉴를 드롭다운 또는 사이드바로 표시해야 함
- **FR-027**: 히어로 섹션은 반드시 모바일에서 텍스트와 이미지가 수직 배치되어야 함
- **FR-028**: Skills 그리드는 반드시 화면 크기에 따라 열 수를 조정해야 함 (데스크톱: 4-6열, 태블릿: 3-4열, 모바일: 1-2열)
- **FR-029**: 모든 텍스트와 이미지는 반드시 화면 크기에 맞게 조정되어 잘리거나 겹치지 않아야 함

**테마 및 접근성**

- **FR-030**: 웹사이트는 반드시 시스템의 다크모드 설정을 감지하여 자동으로 적용해야 함 (prefers-color-scheme)
- **FR-031**: 모든 이미지는 반드시 alt 텍스트를 포함해야 함
- **FR-031-1**: 이미지 로딩 실패 시 반드시 대체 이미지(placeholder)를 표시해야 함
- **FR-032**: 페이지는 반드시 3초 이내에 초기 로딩이 완료되어야 함
- **FR-033**: 섹션 간 스크롤 애니메이션은 반드시 부드럽고 자연스러워야 함

### 주요 엔티티

- **포트폴리오 데이터**: 개발자의 기본 정보를 담고 있는 데이터 구조
  - 속성: 이름(Codeliner), 직책(Front-End Developer), 소개 문구, About Me 텍스트, 프로필 사진 URL, GitHub 프로필 URL
  - 데이터 소스: TypeScript 상수 또는 JSON 파일로 정적 관리 (빌드 타임에 번들링)
- **기술 스택 항목**: 개발자가 사용하는 각 기술을 나타내는 데이터

  - 속성: 기술 이름, 아이콘 이미지 URL(외부 CDN 링크), 숙련도(선택적)
  - 관계: 포트폴리오 데이터에 여러 개의 기술 스택이 포함됨 (1:N)
  - 데이터 소스: TypeScript 상수 배열 또는 JSON 파일로 정적 관리
  - 아이콘 소스: devicons, simpleicons 등 외부 CDN 사용

- **프로젝트**: 개발자가 진행한 사이드 프로젝트를 나타내는 데이터
  - 속성: 프로젝트 이름, 설명, 사용 기술 목록, 프로젝트 이미지 URL, GitHub 링크, 라이브 데모 링크, 시작/종료 날짜(선택적)
  - 관계: 포트폴리오 데이터에 여러 개의 프로젝트가 포함됨 (1:N), 각 프로젝트는 여러 기술을 사용함 (N:M)
  - 데이터 소스: TypeScript 상수 배열 또는 JSON 파일로 정적 관리

## 성공 기준 _(필수)_

### 측정 가능한 결과

- **SC-001**: 방문자는 페이지 로드 후 3초 이내에 모든 핵심 섹션(헤더, 히어로, About, Skills, Contacts)의 콘텐츠를 확인할 수 있어야 함
- **SC-002**: 방문자는 헤더의 네비게이션 링크를 클릭하여 1초 이내에 원하는 섹션으로 이동할 수 있어야 함
- **SC-003**: 방문자는 Projects 섹션의 캐러셀에서 3번의 클릭 이내에 모든 프로젝트를 탐색할 수 있어야 함
- **SC-004**: 웹사이트는 데스크톱(> 1024px), 태블릿(768px - 1024px), 모바일(< 768px) 모든 화면 크기에서 콘텐츠가 읽을 수 있고 잘리지 않아야 함
- **SC-005**: GitHub 프로필 링크를 클릭한 방문자의 90% 이상이 실제로 GitHub 프로필 페이지로 이동해야 함 (유효한 링크)
- **SC-006**: 페이지의 Lighthouse 접근성 점수가 90점 이상이어야 함
- **SC-007**: 페이지 로딩 시간이 느린 네트워크(3G)에서도 10초 이내여야 함
- **SC-008**: 방문자는 모바일 기기에서 햄버거 메뉴를 1번의 탭으로 열고 원하는 섹션으로 이동할 수 있어야 함
- **SC-009**: 캐러셀의 프로젝트 슬라이드 애니메이션이 60fps로 부드럽게 작동해야 함
- **SC-010**: 웹사이트를 방문한 채용 담당자의 80% 이상이 개발자의 기술 스택과 프로젝트 경험을 명확히 이해할 수 있어야 함 (사용자 피드백 기반)
